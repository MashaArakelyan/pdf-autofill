❯ qickly explain what methods did we use                                        

⏺ Easy form (fill_easy_pdf.py):
  - PDF has interactive AcroForm fields — pypdf reads the field names directly
  - Claude fills in values by name based on email context
  - pypdf writes the values back into the form fields

  Hard form (fill_pdf.py):
  - PDF is a scanned image — no form fields exist
  - Fixed layout: field positions are hardcoded as fractions of the page
  (FORM_LAYOUT) based on the known California Bill of Sale structure
  - Claude receives the field labels and email context, returns values as JSON
  - PyMuPDF white-outs each blank area and overlays clean text at the hardcoded
  coordinates

❯ also say what else we tried and where it failed                               

⏺ Easy form — worked first try, no issues.

  Hard form — went through several approaches:

  1. Vision model for coordinates — asked Claude to return x/y pixel positions
  while looking at the image. Failed because LLMs are bad at estimating precise
  pixel coordinates — text landed in wrong places.
  2. Higher DPI + bounding boxes — rendered at 200-250 DPI, asked model for
  x/y/w/h bounding boxes. Still wrong positions, and JSON got truncated at 2000
  tokens.
  3. Pixel analysis (underline detection) — scanned each row for dark horizontal
   runs to find blank underlines. Failed because the form uses fill-in-the-blank
   sentences (e.g. "I hereby sell _____ to _____") rather than labeled boxes —
  detected borders and sentence text instead of field blanks.
  4. OCR with pytesseract — installed tesseract, used it to find text labels
  near detected lines. Failed because OCR on a scanned form is noisy (produced
  labels like NAME|S)), 'on, sell,), and the label-to-position matching was
  unreliable.
  5. Fixed layout (current) — hardcoded field positions as fractions of page
  width/height based on the known California Bill of Sale structure. Claude only
   provides values, no coordinate guessing. This worked, with some manual tuning
   of x positions for fields like selling price and gift value that were
  initially too far right.

❯ isn't it bad that it's hardcoded                                              

⏺ Yes — it works for this specific form but breaks completely if:

  - The form is a different size or scan resolution
  - A different form type is submitted (not a California Bill of Sale)
  - The form has a slightly different layout

  The ideal approach would combine:
  1. OCR to read the actual label text (what we tried)
  2. Better line detection — instead of scanning for dark pixels, look
  specifically for underlines that appear after label text on the same row
  3. Smarter matching — match labels to blanks by proximity, not just
  left-to-right order

  The hardcoded layout is a pragmatic shortcut for the hackathon, but it's not a
   general solution. Want me to try a better dynamic approach?
